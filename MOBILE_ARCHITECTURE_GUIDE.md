# ТЕХНИЧЕСКАЯ СПЕЦИФИКАЦИЯ ПРЕМИУМ-УСЛУГ (MOBILE ARCHITECTURE GUIDELINES)
## Назначение документа
Настоящее руководство предназначено для команды мобильной разработки (iOS/Android/Flutter) и бэкенд-инженеров. Документ содержит исчерпывающее техническое описание всех **21 премиум-услуги** эзотерической платформы «Анти-Таро».

Вся логика перенесена из боевого Python/VKBottle/Supabase-бэкенда и структурирована по **5 архитектурным группам (Groups A-E)**.

---

## ГЛОБАЛЬНЫЕ СТАНДАРТЫ И БЕЗОПАСНОСТЬ (CORE ENGINE)

### 1. Транзакционная безопасность и логика Rollback (Возврат средств)
Каждая операция платного разбора обернута в блок `try/except/finally`. Если в процессе генерации ИИ, обработки Pillow-изображения или компиляции PDF-отчета WeasyPrint происходит сбой (сетевой таймаут `Gemini`, превышение лимитов RPM `429`, ошибка рендеринга), система выполняет автоматический откат баланса:
- Стоимость услуги возвращается на баланс пользователя (`add_energy(vk_id, price)`).
- Пользователю отправляется дружелюбное сообщение с предложением повторить попытку позже:
  > *«🛑 ПОТОКИ КОСМОСА ПЕРЕГРУЖЕНЫ. Звезды скрылись за облаками матрицы. Повтори попытку через минуту, твоя Энергия сохранена и возвращена на баланс.»*
- В мобильном бэкенде эта логика должна быть атомарной (транзакции СУБД) для избежания race-conditions.

### 2. Защита от сленга (Esoteric Slang Dynamic Filter)
Для сохранения благородной, премиальной атмосферы «Анти-Таро» внедрен строгий двухуровневый фильтр сленга:
- **Уровень ИИ**: Системные промпты строго запрещают использовать кибернетические, техно- и хакерские метафоры.
- **Уровень Очистки (clean_slang в modules/utils/pdf.py)**: При сборке PDF все неподходящие фразы заменяются регулярными выражениями с сохранением падежей:
  - `ПЕРЕПРОШИВКА УСПЕХА` $\rightarrow$ `АКТИВАЦИЯ ИЗОБИЛИЯ`
  - `перепрошивка` $\rightarrow$ `трансформация`
  - `прошивка` $\rightarrow$ `судьба`
  - `баг/баги` $\rightarrow$ `урок/уроки`
  - `код/коды` $\rightarrow$ `поток/потоки`
  - `матрица` $\rightarrow$ `космос`
  - `взлом/взломать` $\rightarrow$ `раскрытие/раскрыть`

### 3. Глобальный Tone of Voice (ToV) Blacklist
Модели ИИ категорически запрещено использовать токсичные, обвиняющие или манипулятивные фразы.
**Строжайший бан на фразы**: `хватит ныть`, `разуй глаза`, `ты тормозишь`, `лень`, `прокрастинация`, `пассивность`, а также обращение капсом `МУЖЧИНА`.
Тон должен оставаться уважительным, сакральным, глубоким и дешифрующим.

### ГРУППА A: SIGIL MASTER (САКРАЛЬНАЯ ГЕОМЕТРИЯ)

#### 1. Сигил-Мастер: Генератор знаков (`sigil`)
* **Стоимость**: 1000 Энергии.
* **Описание**: Трансформация намерения/желания пользователя в уникальный графический символ (сигил) по канонам практической магии.
* **Технический сценарий**:
  1. Пользователь списывает 1000 Энергии $\rightarrow$ Состояние FSM переходит в `waiting_sigil_wish`.
  2. Пользователь вводит текст желания (валидация: длина $\ge 5$ символов).
  3. Состояние FSM сбрасывается. Запускается асинхронный процесс генерации.
  4. Бэкенд генерирует Pillow-изображение сигила (см. алгоритм ниже) и отправляет запрос к Gemini.
* **Входные данные**: `wish_text` (текст желания).
* **Gemini Prompt**: `GROUP_A_PROMPT` (принимает `{wish_text}`).
* **Gemini Response JSON-схема**:
  ```json
  {
    "geom_analysis": "Детальный разбор геометрии знака, линий, осей и пересечений.",
    "activation_ritual": "Пошаговая инструкция ментальной активации сигила (медитация, нанесение).",
    "energy_vector": "Направление сил пользователя в реальном мире для материализации.",
    "focus_mantras": ["Мантра 1", "Мантра 2", "Мантра 3"]
  }
  ```
* **Алгоритм Pillow (`generate_sigil_image`)**:
  - Берется базовый фон `cards/uslugi/Новый проект (13).png` (или темно-космический радиальный градиент размером 1080x1080).
  - Вычисляется MD5-хэш от `wish_text`. Хэш переводится в список 16 целых чисел от 0 до 255.
  - На основе чисел хэша вычисляются радиусы трех окружностей: $r_1, r_2, r_3$.
  - Окружности рисуются золотым цветом `(244, 212, 140)`. Рисование происходит на двух слоях: слое резких линий (`sharp_layer`, толщина 3px) и слое неонового свечения (`glow_layer`, толщина 14px).
  - Вычисляется количество вершин звездного глифа $N = 6 + (num_3 \pmod 8)$.
  - Вершины соединяются линиями крест-накрест с шагом $skip = 1 + (num_5 \pmod{N/2})$. На вершинах рисуются рунические круговые засечки.
  - `glow_layer` размывается фильтром `GaussianBlur(15)` и объединяется с базовым фоном и `sharp_layer` методом `Image.alpha_composite`.
  - Снизу накладывается заголовок «СА К Р А Л Ь Н Ы Й   С И Г И Л» (шрифт `Lora-Bold.ttf`, размер 40) и обрезается желание пользователя до 50 символов в кавычках (шрифт `Lora-Regular.ttf`, размер 32).
  - Картинка временно сохраняется на диск, загружается в чат и удаляется.
* **WeasyPrint PDF-шаблон**: `GROUP_A_TEMPLATE`. Использует переменные `sigil_photo` (путь к сгенерированному сигилу) и распакованный JSON ИИ (`geom_analysis`, `activation_ritual`, `energy_vector`, `focus_mantras`).

* **Сценарий**: FSM `waiting_oracle_question` $\rightarrow$ ввод вопроса $\rightarrow$ FSM `oracle_cut` (кнопка сдвинуть колоду) $\rightarrow$ ИИ генерирует расклад на 3 карты Таро.

#### 20. Анти-Таро (`antitaro`)
* **Стоимость**: 500 Энергии.
* **Описание**: Трезвый, честный взгляд на текущую ситуацию без иллюзий.

#### 21. Карта дня (`card_of_day`)
* **Стоимость**: Бесплатно.
* **Описание**: Ежедневный совет для сонастройки с матрицей дня. Начисляется бонус +100 Энергии при ежедневном получении.

#### Вне группы: Микро-инсайт (`micro_insight`)
* **Стоимость**: 100 Энергии.
* **Описание**: Быстрый совет от персонального Проводника на ближайший час.
* **Сценарий**: Прямой текстовый запрос к Gemini на основе натальных данных и тегов без вытягивания карт. Ограничение — строго 2 предложения.

## ИНТЕГРАЦИЯ С БАЗОЙ ДАННЫХ И ХРАНЕНИЕ (DB & MEMORY SYNC)

### 1. Redis как оперативное FSM-хранилище
Все активные состояния диалога записываются в Redis в строгом JSON-формате для защиты от потери сессий при перезапуске бэкенда.
Пример структуры записи состояния:
```json
{
  "step": "waiting_synastry_city",
  "partner_name": "Александр",
  "partner_date": "10.02.1988",
  "partner_time": "18:45"
}
```
Все состояния регистрируются в глобальной карте `state_map` (`database/states.py`). Если состояние отсутствует в маппинге, FSM-контекст сбрасывается.

### 2. Сохранение в readings_history (Supabase / Postgres)
Каждый успешный разбор премиум-услуги сохраняется в массив `readings_history` пользователя в СУБД. Это критично для мобильного приложения, так как позволяет реализовывать раздел **«Гримуар» (Архив разборов)**.

Структура элемента истории:
```json
{
  "title": "Окуломантия",
  "date": "24.10.2024",
  "text": "Полный текст ИИ-анализа...",
  "section": "oculomancy",
  "data": {
    "text": "Полный текст...",
    "eye_photo": "cards/eye_123456.jpeg",
    "eye_url": "https://vk.com/photo..."
  }
}
```
**Важное правило**: при регенерации PDF-отчета из истории («Гримуара»), если Pillow-изображение (сигил или глаз) удалено с сервера для экономии места, бэкенд на лету воссоздает картинку по метаданным из `readings_history` перед компиляцией PDF и сразу после этого очищает диск.

## ШАБЛОН ДЛЯ МОБИЛЬНОГО ЭКРАНА (UI/UX FLOW)

Для любого платного разбора мобильное приложение должно следовать следующей блок-схеме экранов:

```
[Экран Услуги в Каталоге]
          |
          v
[Проверка баланса Энергии] --(Недостаточно)--> [Экран Пополнения/Тарифов]
          |
      (Достаточно)
          v
[Экран Сбора Данных / FSM] (Ввод текста, квиз или загрузка фото ладоней/глаза)
          |
          v
[Экран Сдвига Колоды / Настройки] (Интерактивная кнопка «Сдвинуть колоду»)
          |
          v
[Экран Загрузки (Shimmer/Typing)] (Эффект «Оракул настраивает связь...»)
          |
          v
[Экран Результата] (Текст разбора + Кнопка «Скачать PDF-отчет» + Кнопка «Оценить»)
```

Эта спецификация содержит исчерпывающие данные для развертывания мобильного API на вашем бэкенде.
